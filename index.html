네트워킹 기본

네트워크란 무엇인가?

네트워크는 지리적으로 분산된 장치(컴퓨터, 서버, 라우터 등)들을 서로 연결하여 정보를 교환하고 자원을 공유할 수 있도록 하는 통신 시스템입니다. 예를 들어, 인터넷은 전 세계의 수많은 네트워크들을 연결하는 거대한 광역 네트워크(WAN)의 한 예입니다. 이를 통해 뉴욕과 보스턴, 혹은 서울과 런던처럼 멀리 떨어진 곳에서도 데이터를 주고받을 수 있습니다. 이 섹션에서는 네트워크의 기초와 주요 특성들을 살펴봅니다.

OSI 모델

OSI 모델 개요

OSI(Open Systems Interconnection) 모델은 국제 표준화 기구(ISO)에서 개발한 네트워크 통신을 위한 개념적 프레임워크입니다. 이 모델은 복잡한 네트워크 통신 과정을 7개의 추상적인 계층으로 나누어 설명함으로써, 서로 다른 시스템 간의 상호 운용성을 확보하고 네트워크 기술 개발 및 학습을 용이하게 합니다.

데이터 캡슐화 및 역캡슐화

캡슐화: 송신 측 데이터는 상위 계층에서 하위 계층으로 이동하면서 헤더(및 트레일러)가 추가됩니다.

역캡슐화: 수신 측 데이터는 하위 계층에서 상위 계층으로 이동하면서 헤더(및 트레일러)가 제거됩니다.

OSI 모델의 이점

  • 네트워크 통신의 복잡성을 줄이고 계층별로 나누어 이해하기 쉽게 합니다.
  • 표준화를 통해 여러 제조업체의 장비 간 상호 운용성을 증진시킵니다.
  • 특정 계층의 변경이 다른 계층에 미치는 영향을 최소화하여 기술 개발과 유지보수를 용이하게 합니다.

Cisco 운영체제

Cisco 운영체제의 간략한 역사

  • Cisco는 1984년 라우터로 시작, IOS(Internetwork Operating System) 사용.
  • Catalyst 스위치는 1993년 Crescendo 인수, 초기 CatOS 사용 (현재 미사용).
  • Cisco 방화벽은 1995년 Network Translation의 PIX 방화벽 인수.
  • 이후 스위치와 방화벽은 IOS로 통합.

기타 Cisco 운영체제

  • IOS는 여전히 대부분의 Cisco 기업용 네트워크 장비의 주요 OS.
  • 최근 개발된 다른 OS: NX-OS (Nexus, MDS), IOS-XR (NCS, CRS 등), IOS-XE (ASR1000).
  • 다른 OS들의 CLI는 IOS와 거의 동일.

Cisco 장비 연결

초기 연결 (콘솔)

Cisco 장비는 기본 IP 주소가 없는 경우가 많아 초기 설정 시 콘솔 연결이 필요합니다. 콘솔 케이블(DB9-RJ45 또는 USB-MiniUSB)을 사용하여 장비의 콘솔 포트와 PC를 연결합니다. Putty와 같은 터미널 에뮬레이터를 사용합니다. (Packet Tracer에서는 시뮬레이션으로 대체)

  • 콘솔 연결은 장비 부팅 과정 확인 및 IP 응답 불가 시 문제 해결에도 사용됩니다.

네트워크를 통한 연결 (SSH/Telnet)

일상적인 관리는 SSH(Secure Shell)를 통해 장비의 관리 IP 주소로 접속합니다. Telnet도 지원되지만 보안에 취약하여 권장되지 않습니다. 기업 환경에서는 AAA 서버와 연동하여 보안 로그인을 강화합니다.

Out-of-Band Management: 운영 트래픽과 관리 트래픽을 분리하기 위해 별도의 관리 네트워크를 구성하는 방식입니다.

IOS 네비게이션

명령어 계층 (Command Hierarchy)

  • User Exec mode: `hostname>` (기본 정보 확인)
  • Privileged Exec mode: `hostname#` (모든 정보 확인, 설정 변경 시작. `enable` 명령어로 진입)
  • Global Configuration mode: `hostname(config)#` (전역 설정. `configure terminal` 명령어로 진입)
  • Interface Configuration mode: `hostname(config-if)#` (인터페이스 설정. `interface [타입 슬롯/포트]` 명령어로 진입)
  • `exit`는 한 단계 위로, `end`는 Privileged Exec 모드로 바로 이동.

유용한 CLI 기능

  • 명령어 축약: 고유하게 식별되는 한 짧게 입력 가능 (예: `en` for `enable`).
  • Context Sensitive Help (?): `sh?` (sh로 시작하는 명령어), `show ?` (show 다음 옵션), `show ip ?` (show ip 다음 옵션).
  • 커서 이동: 화살표 키, Ctrl-A (줄 처음), Ctrl-U (줄 삭제).
  • 명령어 히스토리: 위/아래 화살표 키.
  • 출력 제어: Enter (한 줄씩), Spacebar (한 페이지씩), Ctrl-C (중단).
  • Piped Commands (|): `show running-config | include FastEthernet0/0` (특정 문자열 포함 라인만 표시).

IOS 설정 관리

설정 저장 위치

  • Flash: IOS 운영체제 이미지 저장.
  • NVRAM (Non-Volatile RAM): Startup Configuration (시작 설정) 저장. 부팅 시 이 설정을 사용.
  • RAM (Random Access Memory): Running Configuration (현재 실행 중인 설정) 저장. 장비 부팅 시 Startup Config가 RAM으로 로드됨.

설정 저장 및 초기화

  • 명령어는 즉시 적용되지만, 재부팅 시 사라집니다 (Running Config에만 반영).
  • 설정 저장 (영구 보존): `copy running-config startup-config` 또는 `wr` (축약형).
  • 설정 초기화 (공장 초기화): `erase startup-config` 또는 `wr erase` 후 `reload` (재부팅).

전송 계층 개요

주요 기능

  • 호스트 간의 투명한 데이터 전송을 제공합니다.
  • 종단 간(End-to-End) 오류 복구 및 흐름 제어를 담당합니다.
  • 흐름 제어 (Flow Control): 송신측의 데이터 전송 속도를 조절하여 수신측 호스트가 모든 데이터를 처리할 수 있도록 보장합니다.

세션 멀티플렉싱

개념

세션 멀티플렉싱은 호스트가 단일 링크를 통해 동시에 여러 세션을 지원하고 개별 트래픽 스트림을 관리하는 프로세스입니다. 이를 통해 하나의 컴퓨터에서 웹 브라우징(HTTP), 이메일(SMTP), 파일 전송(FTP) 등 여러 네트워크 애플리케이션을 동시에 사용할 수 있습니다.

예: 송신자는 HTTP (포트 80) 트래픽과 SMTP (포트 25) 트래픽을 동시에 수신자에게 보낼 수 있으며, 수신자는 각 포트 번호를 통해 해당 트래픽을 올바른 애플리케이션으로 전달합니다.

계층 4 포트 번호

역할 및 사용

  • 목적지 포트 번호 (Destination Port Number): 상위 계층 프로토콜(애플리케이션)을 식별하는 데 사용됩니다.
    • 예: HTTP는 포트 80, SMTP 이메일은 포트 25를 사용합니다.
  • 출발지 포트 번호 (Source Port Number): 송신자도 계층 4 헤더에 출발지 포트 번호를 추가합니다. 이 번호는 보통 동적으로 할당됩니다.
  • 세션 추적: 출발지 포트 번호와 목적지 포트 번호의 조합은 특정 통신 세션을 추적하는 데 사용될 수 있습니다. (예: 송신자 포트 1500 -> 수신자 포트 80, 응답은 수신자 포트 80 -> 송신자 포트 1500)

TCP (Transmission Control Protocol)

주요 특징

  • TCP와 UDP는 가장 일반적인 계층 4 프로토콜입니다.
  • 연결 지향 (Connection-oriented): 연결이 설정되면 해당 연결을 통해 양방향으로 데이터를 보낼 수 있습니다.
  • 시퀀싱 (Sequencing): 세그먼트가 올바른 순서로 처리되고 누락된 것이 없는지 확인합니다.
  • 신뢰성 (Reliable): 수신 호스트는 송신자에게 확인 응답(acknowledgment)을 보냅니다. 손실된 세그먼트는 재전송됩니다.
  • 흐름 제어 (Flow control): 수행합니다.

TCP 3-Way 핸드셰이크 (Three-Way Handshake)

TCP 연결을 설정하는 과정입니다:

  1. SYN: 클라이언트가 서버에 연결 요청 (SYNchronize) 메시지를 보냅니다.
  2. SYN-ACK: 서버는 클라이언트의 요청을 수락하고, 자신도 연결을 설정하겠다는 SYNchronize-ACKnowledgment 메시지를 보냅니다.
  3. ACK: 클라이언트는 서버의 응답을 확인했다는 ACKnowledge 메시지를 보냅니다. 이로써 연결이 설정됩니다.

송신자 --- SYN ---> 수신자

송신자 <--- SYN-ACK --- 수신자

송신자 --- ACK ---> 수신자

UDP (User Datagram Protocol)

주요 특징

  • 트래픽을 최선형(Best Effort)으로 전송합니다.
  • 비연결 지향 (Connectionless): 호스트 간 핸드셰이크 연결 설정이 없습니다.
  • 시퀀싱 미수행: 세그먼트 순서 보장이나 누락 확인을 하지 않습니다.
  • 비신뢰성 (Unreliable): 수신 호스트가 확인 응답을 보내지 않습니다.
  • 흐름 제어 미수행: 흐름 제어를 하지 않습니다.
  • 오류 감지 및 복구가 필요한 경우 상위 계층에서 처리해야 합니다.

TCP vs UDP 및 주요 포트

선택 기준

  • TCP: 신뢰성이 중요한 트래픽에 주로 사용됩니다 (예: 웹 브라우징, 이메일, 파일 전송).
  • UDP: 실시간 애플리케이션(음성, 영상 스트리밍 등)은 TCP의 추가 오버헤드를 감당하기 어려워 UDP를 사용합니다. 약간의 데이터 손실이 있더라도 속도가 중요하기 때문입니다.
  • 일부 애플리케이션은 TCP와 UDP를 모두 사용할 수 있습니다 (예: DNS).

네트워크 계층 및 IP 헤더

네트워크 계층 (Layer 3)

  • 패킷을 목적지까지 라우팅하고 서비스 품질(QoS)을 담당합니다.
  • 가장 잘 알려진 프로토콜은 IP(Internet Protocol)이며, 이 섹션에서는 IPv4에 중점을 둡니다.
  • IP는 비연결형 프로토콜로, 계층 3에서는 확인 응답(acknowledgment)이 없습니다.
  • 기타 계층 3 프로토콜에는 ICMP(Internet Control Message Protocol), IPSec 등이 있습니다.

IP 주소 체계

  • 계층 3에서 구현되는 논리적 주소 지정 방식입니다.
  • 네트워크 설계자는 IP 주소 체계를 사용하여 전체 네트워크를 더 작은 '서브넷'으로 분할합니다.
  • 이는 성능 및 보안을 향상시키고 문제 해결을 용이하게 합니다.
  • 계층 2 MAC 주소는 단일 플랫 주소 지정 방식을 사용하는 반면, 계층 3에서는 네트워크 간의 논리적 분리가 이루어집니다.

IP 트래픽 유형

주요 IP 트래픽 유형

  • 유니캐스트 (Unicast): 단일 출발지에서 단일 목적지 호스트로 전송되는 트래픽입니다. 대부분의 인터넷 트래픽이 유니캐스트 방식입니다.

    예: 웹사이트 접속, 이메일 송수신.

  • 브로드캐스트 (Broadcast): 단일 출발지에서 동일 서브넷 내의 모든 호스트로 전송되는 트래픽입니다. 라우터는 기본적으로 브로드캐스트 트래픽을 다른 네트워크로 전달하지 않습니다.

    예: ARP 요청, DHCP Discover 메시지.

  • 멀티캐스트 (Multicast): 단일 출발지에서 특정 그룹에 속한 다수의 관심 있는 호스트들에게만 전송되는 트래픽입니다. 유니캐스트로 여러 번 보내는 것보다 효율적입니다.

    예: IPTV, 온라인 게임, 화상 회의.

IPv4 주소와 2진수 변환

IPv4 주소 기본

  • IPv4 주소는 32비트 길이입니다.
  • 일반적으로 4개의 '옥텟(octet)'으로 나누어 점으로 구분된 10진수 형식으로 표현합니다 (예: `192.168.10.15`).
  • 각 옥텟은 8비트 길이이며 (4 * 8 = 32비트), 0부터 255까지의 값을 가질 수 있습니다.
  • IP 주소는 서버, 프린터, 네트워크 장비(라우터, 스위치)에는 수동(정적)으로 설정하고, 일반 PC에는 DHCP를 통해 자동으로 할당하는 경우가 많습니다.
  • 서브넷 간의 논리적 분리를 이해하려면 IP 주소를 2진수로 이해하는 것이 중요합니다.

10진수를 2진수로 변환하기

컴퓨터는 2진수(0 또는 1)로 작동합니다. 각 2진수 자리는 오른쪽부터 1, 2, 4, 8, 16, 32, 64, 128의 값을 가집니다 (8비트 옥텟의 경우).

예시: 10진수 236을 2진수로 변환

1286432168421
11101100

128 + 64 + 32 + 0 + 8 + 4 + 0 + 0 = 236. 따라서 236은 2진수로 `11101100`입니다.

예시: IPv4 주소 `192.168.10.15`를 2진수로 변환

  • `192` = `11000000` (128 + 64)
  • `168` = `10101000` (128 + 32 + 8)
  • `10` = `00001010` (8 + 2)
  • `15` = `00001111` (8 + 4 + 2 + 1)

따라서 `192.168.10.15` = `11000000.10101000.00001010.00001111` 입니다.

서브넷 마스크

서브넷 마스크의 역할

  • 호스트가 목적지 주소가 동일 서브넷에 있는지 다른 서브넷에 있는지 판단하는 데 사용됩니다.
  • 동일 서브넷 내의 통신은 스위치를 통해 직접 이루어집니다.
  • 다른 서브넷으로의 통신은 라우터를 통해 전달되어야 합니다.
  • 서브넷 마스크도 IP 주소처럼 32비트 길이며, 점으로 구분된 10진수 또는 슬래시 표기법으로 표현됩니다.

네트워크 부분과 호스트 부분

  • IP 주소는 서브넷 마스크에 의해 '네트워크 부분(Network Portion)'과 '호스트 부분(Host Portion)'으로 나뉩니다.
  • 서브넷 마스크에서 2진수 '1'로 표시된 부분은 IP 주소의 네트워크 부분을, '0'으로 표시된 부분은 호스트 부분을 나타냅니다.
  • 예: IP 주소 `192.168.10.15` / 서브넷 마스크 `255.255.255.0`
    • IP (2진수): `11000000.10101000.00001010.00001111`
    • 마스크 (2진수): `11111111.11111111.11111111.00000000`
    • 네트워크 부분: `192.168.10`
    • 호스트 부분: `.15`
  • 유효한 서브넷 마스크는 항상 왼쪽부터 연속된 '1'들로 시작하고, 그 뒤에 연속된 '0'들이 옵니다. (예: `11111111.11110000.00000000.00000000`는 유효, `11101101...`은 유효하지 않음)

특별한 주소: 네트워크 주소와 브로드캐스트 주소

  • 호스트 부분의 모든 비트가 '0'인 주소는 **네트워크 주소(Network Address 또는 Network ID)**를 나타내며, 특정 서브넷 자체를 지칭합니다. 이 주소는 일반 호스트에 할당할 수 없습니다. (예: `192.168.10.0/24`)
  • 호스트 부분의 모든 비트가 '1'인 주소는 해당 서브넷의 **브로드캐스트 주소(Broadcast Address)**를 나타내며, 이 주소로 전송된 트래픽은 해당 서브넷의 모든 호스트에게 전달됩니다. 이 주소도 일반 호스트에 할당할 수 없습니다. (예: `192.168.10.255/24`)
  • 따라서 `192.168.10.0/24` 서브넷에서 호스트에 할당 가능한 IP 주소 범위는 `192.168.10.1`부터 `192.168.10.254`까지입니다.

슬래시 표기법 (CIDR Notation)

개념 및 사용

  • 서브넷 마스크는 항상 왼쪽부터 연속된 '1'들로 구성되므로, '1'의 개수를 세어 슬래시(/) 뒤에 숫자로 표현하는 것이 더 편리합니다. 이를 슬래시 표기법 또는 CIDR(Classless Inter-Domain Routing) 표기법이라고 합니다.
  • 예시 1:
    • 서브넷 마스크 `255.255.255.0`은 2진수로 `11111111.11111111.11111111.00000000` 입니다.
    • '1'이 24개 있으므로 슬래시 표기법으로는 `/24`입니다.
    • IP 주소 `192.168.10.15`와 서브넷 마스크 `255.255.255.0`은 `192.168.10.15/24`로 표현할 수 있습니다.
    • 네트워크 주소는 `192.168.10.0/24`로 표현합니다.
  • 예시 2:
    • 서브넷 마스크 `255.0.0.0`은 2진수로 `11111111.00000000.00000000.00000000` 입니다.
    • '1'이 8개 있으므로 슬래시 표기법으로는 `/8`입니다.
    • IP 주소 `10.10.10.15`와 서브넷 마스크 `255.0.0.0`은 `10.10.10.15/8`로 표현할 수 있습니다.
    • 네트워크 주소는 `10.0.0.0/8`로 표현합니다.

IP 주소 할당 개요 및 문제점

인터넷 주소 할당 방식 (초기 설계)

  • 인터넷 IPv4 주소의 전 세계적 관리는 IANA(Internet Assigned Numbers Authority)에서 수행합니다.
  • 초기 설계 의도:
    • 기업이 인터넷 통신을 원할 경우 IP 주소 범위를 신청합니다.
    • 필요한 호스트 수와 향후 확장성을 고려하여 충분한 크기의 주소 범위를 요청합니다.
    • 할당받은 주소를 각 지사의 호스트에 배분합니다.

IPv4 주소 고갈 문제

  • IPv4 설계 당시 인터넷의 폭발적인 성장을 예측하지 못해 주소 공간이 부족하게 되었습니다.
  • 장기적인 해결책은 훨씬 큰 주소 공간을 가진 IPv6입니다.
  • 현재 대부분의 기업 네트워크에서는 NAT(Network Address Translation)와 사설 IP 주소를 사용하여 이 문제를 해결하고 있습니다. (추후 상세 학습 예정)
  • 이후 설명은 모든 호스트가 공인 IP 주소를 가진다는 초기 IPv4 설계를 기준으로 합니다.

클래스 A 주소

특징

  • 매우 많은 수의 호스트를 가진 대규모 네트워크에 할당됩니다.
  • 첫 번째 비트(최상위 비트)는 항상 0으로 설정됩니다.
  • 기본 서브넷 마스크는 /8 (255.0.0.0) 입니다.
  • 유효한 네트워크 주소 범위: `1.0.0.0` ~ `126.0.0.0/8`
  • 이는 126개의 네트워크와 네트워크당 16,777,214개의 호스트를 허용합니다. (실제로는 네트워크 주소와 브로드캐스트 주소를 제외하면 16,777,212개)
  • 예시: `15.0.0.0/8` -> 첫 옥텟 15 (2진수 `00001111`)

예약된 클래스 A 주소

  • `0.0.0.0/8` (0.0.0.0 ~ 0.255.255.255): '이 네트워크(this network)'를 의미하며 예약되어 있습니다. 유효한 호스트 주소로 사용 불가.
  • `127.0.0.0/8` (127.0.0.0 ~ 127.255.255.255): 로컬 컴퓨터 테스트를 위한 루프백(loopback) 주소로 예약되어 있습니다. (예: `127.0.0.1`) 유효한 호스트 주소로 사용 불가.
  • 이러한 예약으로 인해 상당수의 주소가 사용 불가능하게 되었습니다.

서브넷팅

실제 환경에서는 /8로 할당받은 거대한 네트워크를 단일 논리 네트워크로 사용하지 않습니다. 성능 및 보안 문제로 인해 더 작은 서브넷으로 분할하여 사용합니다. 예를 들어, `15.0.0.0/8`을 할당받은 경우, 뉴욕 영업팀에 `15.0.1.0/24`, 회계팀에 `15.0.2.0/24` 등으로 나누어 할당할 수 있습니다. (서브넷팅은 이후 상세 학습 예정)

클래스 B 및 C 주소

클래스 B 주소

  • 중간 규모에서 대규모 네트워크에 할당됩니다.
  • 처음 두 비트는 항상 `10` (2진수)으로 설정됩니다.
  • 기본 서브넷 마스크는 /16 (255.255.0.0) 입니다.
  • 유효한 네트워크 주소 범위: `128.0.0.0` ~ `191.255.0.0/16`
  • 이는 16,384개의 네트워크와 네트워크당 65,534개의 호스트를 허용합니다.
  • 예시: `131.192.0.0/16` -> 첫 옥텟 131 (2진수 `10000011`)
  • 클래스 B 주소도 실제 환경에서는 서브넷팅하여 사용합니다.

클래스 C 주소

  • 소규모 네트워크에 사용됩니다.
  • 처음 세 비트는 항상 `110` (2진수)으로 설정됩니다.
  • 기본 서브넷 마스크는 /24 (255.255.255.0) 입니다.
  • 유효한 네트워크 주소 범위: `192.0.0.0` ~ `223.255.255.0/24`
  • 이는 2,097,152개의 네트워크와 네트워크당 254개의 호스트를 허용합니다.
  • 예시: `195.0.192.0/24` -> 첫 옥텟 195 (2진수 `11000011`)
  • 클래스 C 주소는 그대로 사용되거나 더 작은 서브넷으로 분할될 수 있습니다.

사설 IP 주소 (Private Addresses)

각 클래스에는 사설망에서 사용하도록 예약된 주소 범위가 있습니다. 이 주소들은 공용 인터넷으로 라우팅되지 않습니다. 원래는 인터넷 연결이 없는 폐쇄된 사설 네트워크를 위해 설계되었습니다.

  • 클래스 A 사설 주소: `10.0.0.0` ~ `10.255.255.255` (`10.0.0.0/8`)
  • 클래스 B 사설 주소: `172.16.0.0` ~ `172.31.255.255` (`172.16.0.0/12`)
  • 클래스 C 사설 주소: `192.168.0.0` ~ `192.168.255.255` (`192.168.0.0/16`)

사설 주소와 NAT에 대해서는 이후 강의에서 자세히 다룹니다.

클래스 D, E 및 요약

클래스 D 주소

  • IP 멀티캐스트 주소를 위해 예약되어 있습니다.
  • 처음 네 비트는 항상 `1110` (2진수)으로 설정됩니다.
  • 이 주소들은 호스트에 할당되지 않으며 기본 서브넷 마스크가 없습니다.
  • 유효한 주소 범위: `224.0.0.0` ~ `239.255.255.255`
  • 예시: `227.1.192.5` -> 첫 옥텟 227 (2진수 `11100011`)

클래스 E 주소

  • '실험용 및 향후 사용을 위해 예약된' 주소입니다.
  • 처음 네 비트는 `1111` (2진수)으로 설정됩니다. (실제로는 첫 4비트가 1111이면 240.0.0.0 부터 시작)
  • 이 주소들은 호스트에 할당되지 않으며 기본 서브넷 마스크가 없습니다.
  • 주소 범위: `240.0.0.0` ~ `255.255.255.255`
  • `255.255.255.255`는 '이 네트워크(this network)'의 브로드캐스트 주소로 사용됩니다.
  • 예시: `243.1.192.10` -> 첫 옥텟 243 (2진수 `11110011`)

CIDR (Classless Inter-Domain Routing)

클래스풀 주소 지정의 문제점과 CIDR 도입

  • 기존 클래스풀(Classful) 주소 지정 방식에서는 호스트 수가 254개를 초과하는 회사는 클래스 B 네트워크(`65,534` 호스트)를 할당받아야 했습니다. 이는 실제 필요보다 훨씬 많은 주소를 할당하여 막대한 양의 IP 주소 공간을 낭비하는 결과를 초래했습니다.
  • 이러한 문제를 해결하기 위해 1993년에 CIDR(Classless Inter-Domain Routing, 사이더)이 도입되었습니다.
  • CIDR은 기존 클래스 A, B, C의 고정된 /8, /16, /24 서브넷 마스크 요구사항을 없애고, 네트워크를 더 작거나 큰 단위로 유연하게 분할(서브네팅)하거나 통합(슈퍼네팅)할 수 있게 했습니다.
  • 예: `175.10.10.0/20` 와 같이 슬래시 표기법을 사용하여 네트워크 크기를 다양하게 표현합니다.
  • 이를 통해 기업은 실제 필요에 더 가까운 주소 범위를 할당받아 주소 낭비를 줄일 수 있게 되었습니다.

CIDR과 경로 요약 (Route Summarization)

  • CIDR의 또 다른 중요한 이점은 여러 네트워크 블록을 하나의 요약된 경로로 인터넷에 광고할 수 있다는 것입니다. 이를 경로 요약 또는 경로 집합(Route Aggregation)이라고 합니다.
  • 예: ISP B가 `175.11.0.0/24`부터 `175.11.255.0/24`까지의 256개 /24 네트워크를 가지고 있다면, 이를 인터넷의 다른 ISP(예: ISP A)에게 광고할 때 개별적으로 광고하는 대신 `175.11.0.0/16`이라는 단일 요약 경로로 광고할 수 있습니다.
  • 경로 요약의 이점:
    • 라우팅 테이블 크기 감소: ISP A는 ISP B의 256개 세부 네트워크를 모두 알 필요 없이 단일 요약 경로만 학습하므로 라우팅 테이블 크기가 줄고 라우터 메모리 사용량도 감소합니다.
    • 네트워크 안정성 향상: ISP B 내부의 개별 링크 장애가 ISP A에 직접적인 영향을 미치지 않습니다. 요약 경로는 변경되지 않으므로, 문제 발생 범위가 로컬 네트워크로 제한되고 라우터의 CPU 부하도 줄어듭니다.

서브네팅 개요 및 계산

서브네팅이란?

  • 하나의 큰 네트워크 주소 공간을 여러 개의 작은 네트워크(서브넷)로 나누는 과정입니다.
  • 예: 한 회사가 여러 부서나 지사를 가지고 있을 때, 단일 IP 주소 범위를 할당받아 이를 각 부서/지사별로 더 작은 서브넷으로 나누어 관리할 수 있습니다.
  • 이를 위해 기존 IP 주소의 호스트 부분(host bits)에서 일부 비트를 빌려와 네트워크 부분(network bits)을 확장합니다.
  • 네트워크 주소 경계선은 항상 오른쪽으로 이동하며, 오른쪽으로 갈수록 서브넷의 수는 증가하지만 각 서브넷당 호스트 수는 감소합니다.

서브넷 및 호스트 수 계산

  • 사용 가능한 서브넷 수 계산: `2^서브넷_비트_수`
    • `서브넷_비트_수`는 원래 클래스풀 마스크에서 추가로 빌려온 비트의 수입니다.
    • 예: 클래스 C(/24) 네트워크를 /28로 서브넷팅하면 4비트(28-24)를 빌린 것이므로, `2^4 = 16`개의 서브넷이 생성됩니다.
    • 예: 클래스 B(/16) 네트워크를 /28로 서브넷팅하면 12비트(28-16)를 빌린 것이므로, `2^12 = 4096`개의 서브넷이 생성됩니다.
    • (참고: 과거에는 모든 비트가 0이거나 모든 비트가 1인 서브넷은 사용하지 않아 2를 빼기도 했으나, 현재 Cisco 라우터에서는 `ip subnet-zero` 명령어가 기본 활성화되어 모든 서브넷을 사용할 수 있습니다.)
  • 서브넷당 사용 가능한 호스트 수 계산: `(2^호스트_비트_수) - 2`
    • `호스트_비트_수`는 서브넷 마스크에서 0으로 남은 비트의 수입니다.
    • 2를 빼는 이유: 호스트 부분의 모든 비트가 0인 주소는 네트워크 주소로, 모든 비트가 1인 주소는 브로드캐스트 주소로 사용되어 호스트에 할당할 수 없기 때문입니다.
    • 예: /28 서브넷 마스크를 사용하면 호스트 비트는 4비트(32-28)이므로, `(2^4) - 2 = 16 - 2 = 14`개의 호스트를 가질 수 있습니다.
  • 서로 다른 서브넷에 있는 호스트 간의 통신은 라우터를 거쳐야 합니다.

클래스 C 서브네팅과 VLSM

클래스 C 서브네팅 예시 (`200.15.10.0/24` 기준)

  • /31 (255.255.255.254):
    • 7비트를 빌려와 `2^7 = 128`개의 서브넷 생성.
    • 호스트 비트 1개. 이론상 2개의 주소(0, 1)가 있지만, 네트워크/브로드캐스트 주소 규칙을 엄격히 적용하면 사용 가능한 호스트가 없습니다.
    • 하지만 /31은 점대점(Point-to-Point) 링크에서 네트워크/브로드캐스트 주소 없이 두 개의 호스트 주소만으로 사용하도록 특별히 허용됩니다 (Cisco 라우터에서 지원).
    • 예: `200.15.10.0` - `200.15.10.1`, `200.15.10.2` - `200.15.10.3`, ..., `200.15.10.254` - `200.15.10.255`
  • /30 (255.255.255.252):
    • 6비트를 빌려와 `2^6 = 64`개의 서브넷 생성.
    • 호스트 비트 2개. `(2^2) - 2 = 2`개의 사용 가능한 호스트. 점대점 링크에 주로 사용.
    • 네트워크 주소는 4씩 증가 (마지막 옥텟 기준).
    • 예: `200.15.10.0/30` (호스트: .1, .2 / 브로드캐스트: .3), `200.15.10.4/30` (호스트: .5, .6 / 브로드캐스트: .7) ...
    • CCNA 시험에서는 2개 호스트 필요 시 /31보다 /30을 사용하는 것이 일반적입니다 (별도 지시 없을 시).
  • /29 (255.255.255.248):
    • 5비트를 빌려와 `2^5 = 32`개의 서브넷 생성.
    • 호스트 비트 3개. `(2^3) - 2 = 6`개의 사용 가능한 호스트.
    • 네트워크 주소는 8씩 증가.
    • 예: `200.15.10.0/29` (호스트: .1-.6 / 브로드캐스트: .7), `200.15.10.8/29` (호스트: .9-.14 / 브로드캐스트: .15) ...
  • 기타 클래스 C 서브넷 마스크:
    • `/28` (255.255.255.240): 16개 서브넷, 각 14개 호스트.
    • `/27` (255.255.255.224): 8개 서브넷, 각 30개 호스트.
    • `/26` (255.255.255.192): 4개 서브넷, 각 62개 호스트.
    • `/25` (255.255.255.128): 2개 서브넷, 각 126개 호스트.

VLSM (Variable Length Subnet Masking)

  • 초기 라우팅 프로토콜은 FLSM(Fixed Length Subnet Masking)만 지원하여 모든 서브넷이 동일한 크기를 가져야 했습니다. 이는 주소 낭비를 초래했습니다.
  • 현대의 모든 라우팅 프로토콜은 VLSM을 지원합니다.
  • VLSM은 네트워크 내에서 서로 다른 크기의 서브넷 마스크를 사용할 수 있게 하여, 각 서브넷의 호스트 수 요구사항에 맞게 네트워크 크기를 조절함으로써 IP 주소를 효율적으로 사용할 수 있게 합니다.
  • 예: 14개 호스트가 필요한 서브넷과 62개 호스트가 필요한 서브넷을 동일 네트워크 내에 공존시킬 수 있습니다.

VLSM 설계 예제

요구사항 및 설계 단계

할당된 네트워크: `200.15.10.0/24`

네트워크 구성:

  • 뉴욕 지사: 엔지니어링팀 (28 호스트), 영업팀 (14 호스트)
  • 보스턴 지사: 엔지니어링팀 (28 호스트), 영업팀 (7 호스트)
  • 뉴욕-보스턴 간 점대점 링크 (2 호스트)

설계 고려사항:

  • 가장 큰 세그먼트부터 할당합니다.
  • 주소 공간의 시작 부분부터 차례로 할당합니다.
  • (실무에서는 확장성을 고려하지만, CCNA 시험에서는 문제의 요구사항을 정확히 따릅니다.)

1. 엔지니어링 부서 (각 28 호스트 필요)

  • 28 호스트를 지원하려면 최소 30개 호스트를 제공하는 서브넷 마스크가 필요합니다 (`(2^5) - 2 = 30`). 이는 /27 (255.255.255.224)에 해당합니다.
  • 뉴욕 엔지니어링: `200.15.10.0/27` (네트워크: .0, 호스트: .1-.30, 브로드캐스트: .31)
  • 보스턴 엔지니어링: `200.15.10.32/27` (네트워크: .32, 호스트: .33-.62, 브로드캐스트: .63)

2. 뉴욕 영업 부서 (14 호스트 필요)

  • 14 호스트를 지원하려면 /28 (255.255.255.240) 마스크가 필요합니다 (`(2^4) - 2 = 14`).
  • 다음 사용 가능한 주소 시작: `200.15.10.64`
  • 뉴욕 영업: `200.15.10.64/28` (네트워크: .64, 호스트: .65-.78, 브로드캐스트: .79)

3. 보스턴 영업 부서 (7 호스트 필요)

  • 7 호스트를 지원하려면 6개 호스트를 제공하는 /29는 부족하므로, 14개 호스트를 제공하는 /28 마스크를 사용합니다. (문제에서 가능한 가장 작은 서브넷을 사용하라고 가정) 또는 8개 호스트를 지원하는 /29 (255.255.255.248) 마스크 (`(2^3)-2=6`)는 7개 호스트를 지원하지 못하므로, 다음으로 큰 /28 마스크를 사용해야 합니다. (PDF에서는 /28을 선택했으므로 이를 따릅니다. 만약 6개 호스트였다면 /29가 적합)
  • 다음 사용 가능한 주소 시작: `200.15.10.80`
  • 보스턴 영업: `200.15.10.80/28` (네트워크: .80, 호스트: .81-.94, 브로드캐스트: .95)

4. 뉴욕-보스턴 간 링크 (2 호스트 필요)

  • 2 호스트를 지원하려면 /30 (255.255.255.252) 마스크가 필요합니다 (`(2^2) - 2 = 2`).
  • 다음 사용 가능한 주소 시작: `200.15.10.96`
  • 점대점 링크: `200.15.10.96/30` (네트워크: .96, 호스트: .97, .98, 브로드캐스트: .99)

최종 할당 요약 (라우터 인터페이스 IP 포함)

  • 뉴욕 Eng: `200.15.10.0/27` (라우터 .1)
  • 보스턴 Eng: `200.15.10.32/27` (라우터 .33)
  • 뉴욕 Sales: `200.15.10.64/28` (라우터 .65)
  • 보스턴 Sales: `200.15.10.80/28` (라우터 .81)
  • NY-Boston Link: `200.15.10.96/30` (NY 라우터 .97, Boston 라우터 .98)

대규모 네트워크 서브네팅 예제

예제 1: 클래스 B 주소 (`135.15.0.0/16`)를 /29로 서브네팅

  • /29 서브넷은 호스트 비트가 3개이므로, `(2^3) - 2 = 6`개의 호스트를 가집니다.
  • 클래스 B(/16)에서 /29로 서브네팅하면 네트워크 비트로 13비트(29-16)를 빌려옵니다.
  • 생성 가능한 서브넷 수: `2^13 = 8192`개.
  • IP 주소 `135.15.10.138/29`의 경우:
    • 서브넷 마스크: 255.255.255.248
    • 마지막 옥텟의 블록 크기(Magic Number): 256 - 248 = 8.
    • 네트워크 주소는 8의 배수로 증가: `135.15.10.136/29`
    • 다음 네트워크 주소: `135.15.10.144/29`
    • 브로드캐스트 주소: `135.15.10.143`
    • 유효 호스트 범위: `135.15.10.137` ~ `135.15.10.142`

예제 2: 클래스 A 주소 (`60.0.0.0/8`)를 /28로 서브네팅

  • /28 서브넷은 호스트 비트가 4개이므로, `(2^4) - 2 = 14`개의 호스트를 가집니다.
  • 클래스 A(/8)에서 /28로 서브네팅하면 네트워크 비트로 20비트(28-8)를 빌려옵니다.
  • 생성 가능한 서브넷 수: `2^20 = 1,048,576`개.
  • IP 주소 `60.15.10.75/28`의 경우:
    • 서브넷 마스크: 255.255.255.240
    • 마지막 옥텟의 블록 크기: 256 - 240 = 16.
    • 네트워크 주소는 16의 배수로 증가: `60.15.10.64/28`
    • 다음 네트워크 주소: `60.15.10.80/28`
    • 브로드캐스트 주소: `60.15.10.79`
    • 유효 호스트 범위: `60.15.10.65` ~ `60.15.10.78`

예제 3: 클래스 A 주소 (`60.0.0.0/8`)를 /19로 서브네팅 (3번째 옥텟에서 서브네팅)

  • /19 서브넷은 호스트 비트가 13개(32-19)이므로, `(2^13) - 2 = 8190`개의 호스트를 가집니다.
  • 클래스 A(/8)에서 /19로 서브네팅하면 네트워크 비트로 11비트(19-8)를 빌려옵니다.
  • 생성 가능한 서브넷 수: `2^11 = 2048`개.
  • IP 주소 `60.15.10.75/19`의 경우:
    • 서브넷 마스크: 255.255.224.0 (`11111111.11111111.111`00000.00000000)
    • 서브네팅이 일어나는 3번째 옥텟의 블록 크기: 256 - 224 = 32.
    • 네트워크 주소는 3번째 옥텟에서 32의 배수로 증가: `60.15.0.0/19` (10은 0과 32 사이)
    • 다음 네트워크 주소: `60.15.32.0/19`
    • 브로드캐스트 주소: `60.15.31.255`
    • 유효 호스트 범위: `60.15.0.1` ~ `60.15.31.254`

문제 유형: 최소 6개 네트워크로 `134.65.0.0` (클래스 B) 서브네팅

  • 최소 6개 서브넷을 만들려면 `2^n >= 6`을 만족하는 가장 작은 n을 찾아야 합니다. `2^2=4`, `2^3=8`이므로 3비트를 빌려와야 합니다.
  • 클래스 B의 기본 마스크는 /16입니다. 3비트를 빌려오면 새로운 마스크는 /19 (16+3)가 됩니다.
  • 서브넷 마스크: /19 또는 255.255.224.0
  • 네트워크 주소는 3번째 옥텟에서 32씩 증가: `134.65.0.0`, `134.65.32.0`, `134.65.64.0` 등.
  • 각 서브넷당 호스트 수: `(2^(32-19)) - 2 = (2^13) - 2 = 8190`개.

서브네팅 계산 연습 (3, 4번째 옥텟)

문제 유형

일반적으로 다음과 같은 정보를 계산하는 문제가 출제됩니다:

  • 주어진 IP 주소와 서브넷 마스크에 대한:
    • 해당 서브넷의 네트워크 주소
    • 해당 서브넷의 브로드캐스트 주소
    • 할당 가능한 유효 호스트 IP 주소 범위
  • 특정 수의 서브넷과 호스트 요구사항을 만족하는 네트워크 주소 및 서브넷 마스크 설계.

예제 1: 3번째 옥텟 서브네팅 (`172.19.216.50/23`)

  1. 서브네팅 옥텟 확인: /23이므로 3번째 옥텟에서 서브네팅. (17-24비트 범위)
    172.19.216.50
  2. 네트워크 주소 계산:
    • /23 마스크는 3번째 옥텟에서 `11111110` (254)를 의미. 즉, 3번째 옥텟의 마지막 비트가 호스트 부분.
    • 블록 크기 (3번째 옥텟): 256 - 254 = 2. (또는 23번째 비트의 값 = 2)
    • 3번째 옥텟 값 216을 블록 크기 2로 나누어 가장 가까운 작은 배수를 찾으면 216.
    • 네트워크 주소: `172.19.216.0/23`
    • 다음 네트워크 주소: `172.19.218.0/23`
  3. 브로드캐스트 주소 계산: 다음 네트워크 주소(`172.19.218.0`)에서 1을 뺀 주소.
    브로드캐스트 주소: `172.19.217.255`
  4. 유효 호스트 범위 계산: 네트워크 주소 + 1 부터 브로드캐스트 주소 - 1 까지.
    유효 호스트: `172.19.216.1` ~ `172.19.217.254`

예제 2: 4번째 옥텟 서브네팅 (`172.19.216.50 255.255.255.240`)

  1. 서브네팅 옥텟 확인: 마스크 `255.255.255.240` (/28)이므로 4번째 옥텟에서 서브네팅.
    172.19.216.50
  2. 네트워크 주소 계산:
    • 4번째 옥텟 마스크 값 240.
    • 블록 크기 (4번째 옥텟): 256 - 240 = 16.
    • 4번째 옥텟 값 50을 블록 크기 16으로 나누어 가장 가까운 작은 배수를 찾으면 48 (16 * 3).
    • 네트워크 주소: `172.19.216.48/28`
    • 다음 네트워크 주소: `172.19.216.64/28`
  3. 브로드캐스트 주소 계산: 다음 네트워크 주소(`172.19.216.64`)에서 1을 뺀 주소.
    브로드캐스트 주소: `172.19.216.63`
  4. 유효 호스트 범위 계산: 네트워크 주소 + 1 부터 브로드캐스트 주소 - 1 까지.
    유효 호스트: `172.19.216.49` ~ `172.19.216.62`

연습 문제 (08-05 PDF 기반)

IP 주소 `198.22.45.173/26`에 대해 다음을 계산하시오:

  • 서브넷 마스크 (10진수 표기)
  • 네트워크 주소
  • 브로드캐스트 주소
  • 유효 호스트 주소 범위

풀이

  1. 서브넷 마스크 (10진수): /26은 마지막 옥텟에서 처음 2비트를 빌려옴 (`11`000000).
    2진수: `11111111.11111111.11111111.11000000`
    10진수: `255.255.255.192` (128 + 64 = 192)
  2. 네트워크 주소:
    • 서브네팅은 4번째 옥텟에서 발생.
    • 블록 크기 (4번째 옥텟): 256 - 192 = 64.
    • 4번째 옥텟 값 173. 64의 배수 중 173보다 작거나 같은 가장 큰 값은 128 (64 * 2).
    • 네트워크 주소: `198.22.45.128/26`
    • 다음 네트워크 주소: `198.22.45.192/26`
  3. 브로드캐스트 주소: `198.22.45.191`
  4. 유효 호스트 범위: `198.22.45.129` ~ `198.22.45.190`

사설 IP 주소와 NAT

RFC 1918 사설 IP 주소

  • IETF는 RFC 1918을 통해 공용 인터넷에서 라우팅되지 않는 사설 IP 주소 범위를 지정했습니다.
  • 원래 목적: 인터넷 연결이 필요 없는 내부 호스트 간 통신용. 공인 IP 주소는 유료이므로 비용 절감 효과.
  • 주요 사설 IP 주소 범위:
    • 10.0.0.0 ~ 10.255.255.255 (`10.0.0.0/8`)
    • 172.16.0.0 ~ 172.31.255.255 (`172.16.0.0/12`)
    • 192.168.0.0 ~ 192.168.255.255 (`192.168.0.0/16`)
  • 서로 다른 조직이 동일한 사설 IP 주소 범위를 내부적으로 사용할 수 있습니다 (예: Bank A와 Bank B 모두 `192.168.10.0/24` 사용 가능).

IPv4 주소 고갈과 해결책

  • IPv4 설계자들은 인터넷의 폭발적인 성장을 예측하지 못해 약 43억 개의 주소 공간이 부족해졌습니다.
  • 장기적인 해결책: 128비트 주소 체계를 사용하는 **IPv6** (IPv4보다 약 7.9 x 10^28 배 많은 주소 제공).
  • 단기적인 해결책/현재 상황: **NAT(Network Address Translation)**
    • 조직 내부 네트워크에서는 사설 IP 주소를 사용하고, 인터넷 접속 시 라우터에서 이 사설 IP를 소수의 공인 IP 주소로 변환합니다.
    • 하나의 공인 IP 주소를 여러 내부 호스트가 공유하여 인터넷에 접속할 수 있게 합니다.

오늘날의 네트워크와 사설 IP

  • 2000년대 초반 IPv6로의 빠른 전환이 예상되었으나, 대부분의 기업은 여전히 RFC 1918 사설 IPv4 주소와 NAT를 주로 사용합니다.
    • 이유: 내부 호스트를 숨기는 보안 이점, IPv4에 대한 엔지니어들의 높은 숙련도.
  • IPv6는 주로 서비스 제공업체, 모바일 서비스, 인터넷 도입이 늦었던 대규모 국가(인도, 중국 등)에서 많이 사용됩니다.
  • 공인 IPv4 주소는 2011년에 고갈되어 IPv6로의 전환은 여전히 미래의 과제입니다.
  • 현대 기업들은 성능 및 보안 최적화를 위해 RFC 1918 사설 주소 공간 내에서도 서브네팅을 수행합니다.
  • 일반적으로 종단 호스트에는 /24, 점대점 링크에는 /30, 루프백 인터페이스에는 /32 서브넷을 사용하는 경우가 많습니다.
  • 공인 IP를 내부망에 사용하는 경우 주소 절약을 위해 복잡한 VLSM이 더 흔하게 사용됩니다.

연속적인 주소와 경로 요약

  • 네트워크 설계 시, 지리적 위치나 부서별로 연속적인(contiguous) IP 주소 블록을 할당하면 경로 요약에 유리합니다.
  • 예: Region A에 `10.0.0.0/24` ~ `10.0.255.0/24`를 할당하고, Region B에 `10.1.0.0/24` ~ `10.1.255.0/24`를 할당하면, 각 지역을 대표하는 `10.0.0.0/16`과 `10.1.0.0/16`으로 요약하여 라우팅 정보를 교환할 수 있습니다.
  • 비연속적인(non-contiguous) 주소 할당은 효과적인 경로 요약을 어렵게 만들어 라우팅 테이블을 비대하게 만들 수 있습니다.

데이터 링크 계층 개요

주요 기능 (OSI 모델 계층 2)

  • 프레임(Frame)을 비트(Bit)로 인코딩하고 디코딩합니다.
  • 물리 계층(Layer 1)에서 발생할 수 있는 오류를 감지하고 수정하는 기능을 제공할 수 있습니다.
  • 물리적 주소 지정(MAC 주소)을 담당합니다.
  • 네트워크 인터페이스 간의 데이터 전송을 제어합니다.

이더넷 (Ethernet)

  • 로컬 영역 네트워크(LAN)에서 가장 널리 사용되는 계층 2 기술입니다.
  • CSMA/CD (Carrier Sense Multiple Access with Collision Detection)라는 매체 접근 제어 방식을 사용합니다 (전통적인 공유 이더넷 환경).
  • 스위치 환경에서는 각 포트가 별도의 충돌 도메인을 가지므로 CSMA/CD의 영향이 줄어듭니다.

이더넷 헤더 및 MAC 주소

이더넷 프레임 구조 (Ethernet II 기준)

이더넷 프레임은 다음과 같은 주요 필드로 구성됩니다:

Preamble7 Bytes
SFD1 Byte
Destination MAC6 Bytes
Source MAC6 Bytes
EtherType2 Bytes
Data (Payload)46-1500 Bytes
FCS4 Bytes
  • Preamble (프리앰블): 7 바이트. 프레임의 시작을 알리고 동기화를 위한 비트 패턴 (`10101010` 반복).
  • SFD (Start Frame Delimiter): 1 바이트. 프레임 시작 구분자 (`10101011`).
  • Destination MAC Address (목적지 MAC 주소): 6 바이트. 프레임을 수신할 장치의 MAC 주소.
  • Source MAC Address (출발지 MAC 주소): 6 바이트. 프레임을 송신하는 장치의 MAC 주소.
  • EtherType (이더타입) / Length: 2 바이트. 상위 계층 프로토콜을 나타내거나(Ethernet II), 페이로드의 길이를 나타냅니다(IEEE 802.3). 예: IPv4는 `0x0800`, ARP는 `0x0806`.
  • Data (데이터) / Payload (페이로드): 46 ~ 1500 바이트. 상위 계층에서 전달된 데이터 (예: IP 패킷). 최소 길이를 맞추기 위해 패딩이 추가될 수 있습니다.
  • FCS (Frame Check Sequence): 4 바이트. 프레임 오류 검사를 위한 CRC(Cyclic Redundancy Check) 값.

MAC 주소 (Media Access Control Address)

  • 이더넷에서 사용되는 48비트(6바이트)의 물리적 주소입니다. 16진수로 표현됩니다 (예: `00:50:56:C0:00:08`).
  • 전 세계 모든 네트워크 인터페이스 카드(NIC) 포트에 고유하게 할당된 주소입니다 (Burned-In Address, BIA).
  • OUI (Organizationally Unique Identifier): 앞쪽 24비트(3바이트). IEEE가 NIC 제조업체에 할당하는 고유 식별자입니다.
  • Vendor Assigned (제조업체 할당 부분): 뒤쪽 24비트(3바이트). 해당 제조업체가 자체적으로 할당합니다.
  • 48비트 주소 공간은 이론적으로 $2^{48}$ (약 281조)개의 고유한 MAC 주소를 가질 수 있습니다.

물리 계층 개요

주요 기능 (OSI 모델 계층 1)

  • 비트 스트림(전기적 임펄스, 광 신호 또는 무선 신호)을 네트워크를 통해 전기적 및 기계적 수준에서 전달합니다.
  • 케이블, 인터페이스 카드 및 물리적 측면을 정의하는 등 데이터를 송수신하는 하드웨어 수단을 제공합니다.

UTP 케이블

이더넷용 UTP 케이블

  • 이더넷 LAN 연결은 동축 케이블(더 이상 사용되지 않음), 꼬임쌍선 구리 케이블, 광섬유 케이블 또는 무선으로 이루어질 수 있습니다.
  • 구리 UTP(Unshielded Twisted Pair) 케이블은 데스크톱 컴퓨터를 스위치에 연결하는 데 일반적으로 사용됩니다.
  • 커넥터 유형은 RJ-45이며 최대 길이는 100미터입니다.

스트레이트-스루(Straight-Through) vs 크로스오버(Crossover) UTP 케이블

  • UTP 케이블의 수신 및 송신 와이어는 RJ-45 커넥터에 스트레이트-스루 또는 크로스오버 방식으로 연결될 수 있습니다.
  • 스트레이트-스루 케이블: PC나 라우터와 같은 종단 장치를 스위치에 연결하는 데 사용됩니다.
  • 크로스오버 케이블: 장치를 직접 연결하는 데 사용됩니다. 주로 동일한 유형의 두 장치(예: 두 대의 컴퓨터 또는 두 대의 스위치)를 서로 연결하는 데 사용됩니다.
  • 최신 스위치는 Auto MDI-X를 지원하여 수신 및 송신 신호가 자동으로 재구성되어 예상되는 결과를 얻을 수 있도록 합니다.

광섬유 케이블

특징 및 용도

  • 더 긴 거리나 더 높은 대역폭 요구 사항을 지원하는 데 사용할 수 있습니다.
  • 예: 캠퍼스 내 별도 건물 간 연결 또는 건물 내부 스위치 간 연결.

(광섬유 케이블 이미지 예시 위치)

싱글 모드(Single Mode) vs 멀티 모드(Multi Mode) 광섬유

  • 싱글 모드 광섬유: 더 높은 대역폭과 더 긴 거리를 지원하지만 비용이 더 비쌉니다. 코어 직경이 매우 작아 빛이 단일 경로로 전파됩니다.
  • 멀티 모드 광섬유: 싱글 모드보다 저렴하지만 지원 거리와 대역폭이 낮습니다. 코어 직경이 더 커서 여러 빛 경로(모드)를 허용합니다.

다양한 유형의 광섬유 커넥터가 사용됩니다. (예: LC, SC, ST 등)

(다양한 광섬유 커넥터 이미지 예시 위치)

PoE (Power over Ethernet)

개념 및 활용

  • PoE는 이더넷 케이블을 통해 데이터와 전력을 동시에 전송하는 기술입니다.
  • IP 전화기, 무선 액세스 포인트(AP), IP 카메라 등 별도의 전원 공급 장치가 필요 없는 장치에 유용합니다.
  • PoE를 지원하는 스위치(PoE 스위치)를 사용하거나, 기존 스위치와 장치 사이에 PoE 인젝터(Injector)를 사용하여 구현할 수 있습니다.
